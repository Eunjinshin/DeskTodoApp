# 리팩토링 및 코드 품질 개선 보고서

## 개요
- **목적**: 기존 식별된 11개의 문제점(보안 런타임 에러, 성능, 가독성, 클린 코드 측면)을 개선하고 보다 안정적이고 유지보수하기 쉬운 구조를 확립.
- **방향성**: 클린 코드 원칙 준수, 중복 코드 제거, SQL 인젝션 취약점 대응, 적절한 에러 핸들링 및 로깅 적용.

---

## 🔴 해결된 핵심 문제 (Critical)

### 1. `audit_log` 런타임 에러 해결 (P1)
- **Problem**: 기존 DB에는 `todo_id`로 되어있던 감사 로그 컬럼을 새 코드에서는 `target_id`로 삽입하려다 SQL 에러(런타임 에러) 발생. 앱 구동 불가의 원인.
- **Solution**: `schema.js`의 `MIGRATIONS` 배열에 `ALTER TABLE audit_log RENAME COLUMN todo_id TO target_id` 및 `target_type` 컬럼 추가 구문을 병합. 기존 DB와 신규 코드 간 불일치 해소.
- **결과**: 앱 실행 시 런타임 에러가 소멸되며, 모든 TODO/Schedule 변경 사항이 정상적으로 로깅됨.

### 2. SQL Injection 위험 완화 (P2)
- **Problem**: `init.js`에서 SQLCipher의 `PRAGMA key`를 설정할 때 템플릿 리터럴로 `passphrase`를 직접 조립하여 Injection 위험 존재.
- **Solution**: SQLCipher의 PRAGMA 구문 특성상 Prepared Statement를 직접 사용할 수 없으므로, 싱글 쿼트(`'`)를 이스케이프(`''`) 처리하는 유틸리티(`escapePassphrase`) 적용.

## 🟡 성능 및 구조적 개선 (Medium)

### 3. Window 제어 IPC 핸들러 중복 로직 제거 (P4)
- **Problem**: 창 최소화, 닫기, 핀(pin) 고정, 투명도 조절 등 5개의 핸들러가 각각 `BrowserWindow.fromWebContents(event.sender)`를 중복 호출함.
- **Solution**: 해당 코드를 공통 헬퍼 메서드인 `getWindow(event)`로 추출하여 가독성과 유지보수성을 극대화함.

### 4. 입력 검증 오탐지(False Positive) 방지 (P5)
- **Problem**: 기존 `validator.js`는 `SELECT` 등 SQL 키워드가 포함된 일반 문자열(예: "select item")까지 SQL Injection으로 간주하고 모두 차단.
- **Solution**: 이미 쿼리 실행단에서 Prepared Statement(`?`)를 완벽하게 사용하고 있으므로, 과도한 정규식 기반 키워드 필터링은 삭제하여 정상적인 사용자 입력 보장.

### 5. 상태 동기화 불일치 개선 (P6)
- **Problem**: TODO 생성 시 프론트엔드는 `ADD_TODO` 액션으로 새 항목을 **무조건 최상단**에 렌더링하지만, DB는 `priority DESC`(상 우선)로 정렬되어 다음 조회 시 목록 순서가 뒤섞이는 현상.
- **Solution**: `addTodo` 완료 직후 리듀서를 통한 단순 객체 삽입 대신, `loadTodos()`를 재실행하여 매 순간 DB의 정확한 정렬 순서를 화면과 동기화.

## 🟢 클린 코드 및 유틸성 강화 (Low)

### 6. API Proxy의 불필요한 중복 제거 (P8)
- **Problem**: `todoApi.js`, `scheduleApi.js`, `windowApi.js` 3개 파일에서 `window.secureAPI` 유무를 확인하는 `getAPI()` 함수가 모두 개별 선언됨. 
- **Solution**: 프론트엔드의 `api/base.js`를 신설하여 `getAPI()`를 공용화, DRY(Don't Repeat Yourself) 원칙 준수.

### 7. 누락된 입력 스키마 규격화 (P9)
- **Problem**: 2차 보완 때 추가된 `tag` 필드는 프론트엔드 XSS 대응(`sanitize.js`)은 적용됐으나, 백엔드 IPC 수신부(`validator.js`)에는 제한 규칙이 없음.
- **Solution**: `TODO_SCHEMA`에 `tag` 문자열 제약(최대 30자) 추가.

### 8. 마이그레이션 실패 시 에러 로깅 강화 (P10)
- **Problem**: `schema.js`의 `MIGRATIONS` 단계에서 이미 추가된 컬럼(`duplicate column` 등) 에러를 무시하는 로직이 허술하게 짜여있어, 실제 문제가 될 수 있는 에러조차 은폐될 위험.
- **Solution**: 안전하게 무시해야 할 에러(`duplicate`, `no such column`)만 걸러내고, 예상치 못한 SQL 에러는 `console.warn()`으로 남기도록 개선.

### 9. 불필요 인자 정리 (P7)
- **Problem**: `schedule:list` 호출 시 프론트엔드에서는 전체를 불러오는데 `queries.js` 호출 인자로 `undefined` 두 개를 넘겨주고 있었음. 
- **Solution**: 사용되지 않는 인자를 깔끔하게 정리.

---

## 결론
리팩토링을 통해 백엔드 런타임 오류로 앱이 구동되지 않는 Critical 버그가 말끔히 해소되었습니다. 프론트엔드와 백엔드를 잇는 IPC 통신의 유효성 검증과 정렬 일관성 또한 향상되어 시스템 전체의 **신뢰성과 보안성**이 한층 강화되었습니다.
